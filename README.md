# RestaurantN Mobile Client (Diploma Demo)

Клиентское мобильное приложение ресторана на `React Native + Expo` с интеграцией в командный backend (Syncflow API).  
Текущий репозиторий демонстрирует **пример внедрения в один конкретный ресторан (RestaurantN)**.

## Product Concept

- Платформа проектируется как **white-label решение** для ресторанного бизнеса.
- В демо используется один бренд (`RestaurantN`), один адрес и один набор данных.
- Для реального внедрения под нового заказчика настраиваются:
  - название приложения;
  - логотип и иконка;
  - цвета и тексты интерфейса;
  - контент меню и бизнес-правила в пределах API.

## What Is Implemented

### Client-side UX

- Авторизация/регистрация гостя.
- Меню с категориями, поиском, избранным, детальной карточкой блюда.
- Раздел «Сегодня рекомендуем» с плавной каруселью и индикатором.
- Корзина с валидацией, промокодом, баллами и типом заказа.
- Бронирование стола (дата, время, гости, выбор стола).
- История бронирований, экран деталей брони.
- История заказов/доставок, уведомления, профиль.
- Единый адрес ресторана (single-location модель для малого бизнеса).

### Business logic already wired

- Сессия и хранение токенов.
- Loyalty flow: баланс, списание/проверки ограничений на клиенте.
- Промокод flow на стороне API.
- Обработка ошибок API через user-friendly тексты.
- Кэширование данных (`AsyncStorage`) для устойчивости интерфейса.

### Detailed Feature Behavior (for demo and thesis)

- Экран входа:
  - один сценарий для входа и регистрации;
  - валидация обязательных полей перед отправкой;
  - понятные сообщения об ошибках вместо технических stack/error codes.
- Экран меню:
  - поиск по названию блюда с debounce;
  - категории «Все»/предметные/«Избранное»;
  - «Сегодня рекомендуем» в шапке списка (не закреплено), единый вертикальный скролл;
  - плавный animated-индикатор карусели.
- Карточка блюда:
  - описание и состав;
  - догрузка ингредиентов и модификаторов из API, fallback-текст при серверной ошибке;
  - добавление в корзину и блок рекомендаций.
- Корзина:
  - изменение количества позиций;
  - применение промокода;
  - включение/выключение списания баллов;
  - расчёт лимита списания и финальной суммы;
  - сценарии: бронь, самовывоз, доставка.
- Самовывоз:
  - единая точка выдачи `RestaurantN` без выбора филиала;
  - минимизация пользовательских шагов и отсутствие лишних полей.
- Доставка:
  - расширенный адресный блок (город, улица, дом и доп. параметры);
  - предвалидация обязательных полей до API-вызова.
- Бронирование:
  - выбор числа гостей, даты, времени, времени подачи;
  - фильтрация доступных столов по вместимости и интервалу;
  - передача данных брони в API;
  - детальный экран конкретной брони.
- Профиль:
  - сведения о пользователе;
  - баланс программы лояльности;
  - быстрый вход в заказы, брони, уведомления.
- Уведомления/история:
  - отдельные экраны с пагинацией;
  - восстановление данных из кэша при кратковременных сетевых сбоях.
- Надёжность:
  - refresh access token через `syncflowHttp`;
  - не-блокирующие fallback-механики для read-сценариев;
  - единый слой маппинга backend-ошибок в UX сообщения.

## Current Backend Status (Important)

В интеграции используются API-методы из `API_DOCS.md`, но часть клиентских сценариев пока ограничена серверной реализацией:

- для некоторых методов backend может возвращать `500` (например, столы/ингредиенты на текущем стенде);
- заказ из клиентского приложения (самовывоз/доставка) сейчас зависит от серверного контракта и в части потока ограничен;
- часть возможностей уровня enterprise (multi-tenant DB, push pipeline, white-label provisioning panel) описана как roadmap.

## Architecture (Client)

- `App.js` — composition root и провайдеры.
- `src/navigation/AppNavigator.js` — маршрутизация, сценарии экранов, toast/modal orchestration.
- `src/contexts/AuthContext.js` — auth session lifecycle.
- `src/contexts/ClientDataContext.js` — меню, столы, брони, заказы, профиль, уведомления, локальный кэш.
- `src/services/api/*` — адаптеры API (`syncflow`, `local`, `mock`).
- `src/services/syncflowHttp.js` — HTTP layer, refresh token flow, user message mapping.
- `src/screens/*` — feature screens.
- `src/components/*` — UI-блоки и reusable widgets.
- `src/constants/venue.js` — брендовые константы RestaurantN.

## Configuration

Основной runtime-конфиг задаётся в `app.json`:

- `expo.name`: `RestaurantN`
- `expo.slug`: `restaurantn-client`
- `expo.extra.integratedBackend`: `syncflow` или `local`
- `expo.extra.apiBaseUrl`: URL API
- `expo.extra.useMockApi`: `true/false`

## NFR (Reliability, Security, Scalability)

### Reliability

- Таймауты и контролируемый retry в HTTP-слое.
- Refresh токенов для гостевого канала с повторной попыткой защищённого запроса.
- Локальный кэш (`AsyncStorage`) для read-сценариев и более устойчивого UX при кратковременных сбоях сети.
- User-friendly сообщения об ошибках вместо технических кодов/трассировок.
- Разделение read/write стратегий: для мутаций нет скрытого fallback в mock, чтобы не допускать расхождения источников данных.

### Security

- Авторизация через JWT и защищённые guest-endpoints.
- Минимизация технических деталей ошибок в UI.
- Валидация входных данных до API-вызова на клиенте (обязательные поля, диапазоны, форматы).
- Логическая изоляция ролей и экранных сценариев по контексту пользователя.
- Подготовленная точка расширения под production hardening: device registration + полноценный push pipeline.

### Scalability

- White-label подход (бренд/иконка/цвета/тексты) без изменения core-логики.
- Компонентная архитектура экранов и сервисного слоя.
- Единые адаптеры API и мапперы, упрощающие подключение новых backend-контрактов.
- Roadmap к multi-tenant модели (tenant isolation и provisioning).
- Готовность к переходу от single-location демо к multi-location сценариям.

## Known Gaps + Mitigation

- `GET /api/tables` и часть guest-endpoints на текущем стенде могут возвращать `500`.
  - Mitigation: в UI добавлены диагностические user-facing сообщения и fallback на кэш/предыдущие данные, где это допустимо.
- Детальный состав блюда (`ingredients/modificators`) может быть недоступен при серверной ошибке.
  - Mitigation: карточка блюда показывает описание и понятный fallback-текст без «технического шума».
- В guest-канале не весь order lifecycle доступен на сервере в production-ready виде.
  - Mitigation: в приложении явно разделены поддерживаемые сценарии и ограниченные сценарии, без скрытых «тихих» провалов.
- Push в текущем демо не является полным FCM/APNS pipeline.
  - Mitigation: заложен контракт событий/уведомлений и навигации, что позволяет быстро подключить боевую доставку.
